# Customising Languages in osFinancials5 / TurboCASH5

The **Customise Languages** module offers full flexibility to adapt your financial system to specific industries, corporate preferences, and global localization requirements. By tailoring system terminologies, database classifications, and regional settings, you can ensure compliance with local legislation and seamlessly match your industry's daily workflow.

## Options for Customisation

1. **Editing Labels and Terminologies**: Modify the static text elements used across standard screen layouts. For example, you can globally change the default term "Stock" to "Inventory", or update button text, menu items, and business process labels to fit your industry jargon.
1. **Customising Document Types (The Types Tab)**: Adapt transaction categories to your regional business context. While the core system defaults to *Invoices, Credit notes, and Quotes* for sales, and *Purchases, Supplier returns, and Orders* for procurement, you can rename these categories—such as changing "Quote" to "Estimate".

**The Formatting Rule:** All document name entries within the **Types** tab must be written in a strict **singular format** (Invoice, Credit note, Quote) to ensure dynamic print expressions compile cleanly.

3. **Localising for Country-Specific Environments**: Select or build language matrices that match your country's official language or your employees' linguistic preferences. This enhances overall user experience, clarity, and processing speed.
3. **Tax Regime Alignment**: Configure localized tax descriptions, reporting frameworks, and compliance rules based on your specific regional environment, such as **Value Added Tax (VAT)**, **Goods and Services Tax (GST)**, or standard **Sales Tax** structures.

# What is Affected by Customising the Language?

To maintain absolute data integrity, your localization adjustments cascade systematically across three core layers of the application:

1. [**User Interface (UI)**](<osfinancials5-program.md>): The customized strings instantly populate interactive user interface, including menu options, input field headers, action buttons, and navigation streams (such as the **Documents** on the **Default** ribbon).
1. [**Management Reports**](<reports-ribbon-options.md>): All analytical reports generated by the software reflect your localized terminology. Report headers, table columns, and grouping categories automatically pull from your active language matrix to ensure consistency.
1. **Printed Documents and Layout Templates**: Your customized terms are actively processed by the system's external printing architecture. When a document is generated, the layout file reads the **singular value** from the **Types tab** to print the official document name header.
- &nbsp;

  - **Supported Layout Files:** This synchronization applies across all translatable output formats, including standard *Layout file, Document layout, Document layout (15-code) templates, Delivery note,* and *Point-of-Sale (POS) slip*.
  - **Document Types Renaming in the TYPES Table:** Document layout files such as the "*Layout file, Document layout ,&nbsp; Document layout&nbsp; (15-codes)"*&nbsp; and the *"Delivery note" files* are designed to be versatile and [support various types of documents](<document-layout-files-translate.md>) such as sales documents (invoices, credit notes, quotes and Delivery notes) as well as purchase documents (purchases, supplier returns and orders). Changing the document types in **Tools → [Customise language**](<customise-language.md>) **- Types** tab on the **Setup** ribbon, will print as the Document name when these documents are printed. For example, when you make changes to document types, such as renaming "*Quotes*" to "*Estimates*," these changes will be automatically reflected in the respective document layouts.

| ![Image](<lib/admon-important.png>) | ***TYPES table - Document types must be in the Singular format*** *When changing or localizing document types within the **TYPES table**, you **must** enter the descriptions using a strict **singular format** (e.g., use Invoice, Credit note, Quote, Delivery note, Purchase, Supplier return, Order instead of Invoices, Credit notes, Quotes, Delivery notes, Purchases, Supplier returns, Orders).* ***Why this is mandatory:** If you enter a plural word in the Types table (like "Invoices" or "Quotes"), the software will pull that exact plural term when compiling a single document layout. As a result, an individual customer's printed document will literally display the plural header **"Invoices"** or **"Quotes"** at the top of their sheet, creating an unprofessional and incorrect layout. Keep the entries singular so that a single printed document reads as a single document.* |
| --- | --- |


By ensuring that your customized terms flow directly from the **Types tab** straight onto the physical client-facing print layouts, the system prevents layout fragmentation, ensuring absolute consistency, accuracy, and professional branding across all external business communications.

By having consistent language across the entire osFinancials/TurboCASH system, you create a more cohesive and efficient financial management experience. Customising the language to align with your industry's terminology and business practices allows your team to work seamlessly with the software, leading to improved productivity and accuracy in financial transactions and reporting.

# How to set your customised language file as default?

In osFinancials5/TurboCASH5, you have the option to set or change the default language or your own customized language for the Set of Books through the [**Switch language**](<switch-language.md>) option on the **Start** ribbon.&nbsp; This enables you to have your preferred language automatically loaded whenever you open the Set of Books. After selecting the desired language, save the changes. osFinancials5/TurboCASH5 will now use the selected language as the default whenever you open the Set of Books.

By setting your preferred language or customising the language settings through the [**Switch language**](<switch-language.md>) option on the **Start** ribbon, osFinancials5/TurboCASH5 will display all labels, terminologies, and interface elements, reports and document layout files in the selected language whenever you access your Set of Books. This makes it more convenient and user-friendly, especially when you have customized the language to suit your industry or regional requirements.

# How to set language files as default for users?

Users in **Access control** (**Setup → [Access control**](<set-default-language.md>) menu on the **Setup** ribbon) can have their language set in the "*System lang*" field. Upon logging into osFinancials5/TurboCASH5, the user interface and reports will be displayed in the language assigned to that user.&nbsp;

# Important notes on Customising language files

The language files in osFinancials5/TurboCASH5 are stored in the *\*.dfm* file type, and they can be found in the "*...\\Bin\\Languages*" directory or another directory where osFinancials5/TurboCASH5 is installed. When you first launch osFinancials5/TurboCASH5 after installation, it will automatically display the default language based on the country and language of the Set of Books you selected to open.

However, if you want to use a different language, you can switch to any of the available languages by going to [**Switch language**](<switch-language.md>) in the **Start** ribbon. This allows you to choose from a list of supported languages, including English for various countries, Afrikaans, and others.

# Coordinating Your Chart of Accounts Template with Your Language File

Before starting your language customization workflow, you should establish your underlying financial database structure by selecting or downloading a regional template.

## Selecting or Downloading a Template

**Local Repository**: Open an existing Chart of Accounts template directly from your local ...\\bin\\Repository\\ installation folder.

**Online Download**: Alternatively, utilize the built-in 'Let TurboCASH5-4 help you to create a Set of Books' wizard and select the Download option to pull the latest compliant country-specific template from the osFinancials website.

## Initializing Your Unique Set of Books

Once you locate a suitable accounting template that matches your target country or industry rules, save it under your own unique business name by navigating to the **Start** ribbon and selecting the **Save as** option.

## Setting Your Custom Language as the Default

After creating and registering your new Set of Books database file, you can permanently lock in your localized user interface settings:

* Go to the **Start** ribbon and select the [**Switch language**](<switch-language.md>) option.
* Choose your newly customized language file from the available selection list.

**System Behavior:** Once this link is established, the application will lock this language assignment to the active database. The software will automatically load your customized language file as the default option every time you open or launch this specific Set of Books in the future.

# Spreadsheet Export/Import

For large-scale localization tasks, you can process language files externally using standard spreadsheet applications via a Tab-Delimited environment:

1. Click **ExportFromTab** to generate a clean, external text translation file.
1. **Verify Export Integrity:** Before editing, open the file in your spreadsheet editor (such as OpenOffice or Excel) and immediately check that the total number of records/rows matches exactly **4,072 lines**.

| ![Image](<lib/admon-note.png>) | ***Why this check is necessary**: Some complex text strings or multiline descriptions may accidentally split and export into more than one row depending on your spreadsheets default carriage-return parsing. Catching this early prevents indexing alignment shifts. Some examples may be experienced in long tabel ID's 3276 and 3711.* |
| --- | --- |


3. **Perform your string translations or global re-branding adjustments within the spreadsheet columns:**
- &nbsp;

  - 🔴 **Critical Name Branding Audit:** You **must** verify that the **45 core language labels** containing the \<@APPNAME@\> dynamic placeholders remain completely intact. Do not replace these tags with static text (such as hardcoding "TurboCASH" or "osFinancials") so the application can dynamically brand its UI elements based on the running environment wrapper.
  - 🌐 **URL Source-Match Update:** For the **12 asset labels** referencing system links, ensure that the active URLs exactly mirror the English source text entries (e.g., matching target web paths like \[https://www.osfinancials.org/\](https://www.osfinancials.org/)... or regional domain pointers). If the master English source column specifies a direct URL rather than the legacy \<@APPURL@\> tag, update your translation column to match that exact web string.

4. **Verify Import Integrity:** Once you have finished editing, localizing, or translating your language file in the spreadsheet, **check the row count one final time**. Confirm that the file still contains exactly **4,072 records** before moving to the next step.

4. Return to the software window and click **ImportFromTab** to pull the updated string values back into your language matrix.

# System String Range Indicators

Rather than being locked or restricted, these high-number ID blocks are **functional zone mappings**. They tell you exactly which module or user interface element reads those specific text strings, allowing you to localize complex add-ons and setup fields with precision:

## Plugin \& Layout Context Zones

* **&#57;01000 Block:** Injects text strings directly into the **Subscriptions** plugin interface.
* **&#57;02000 Block:** Injects text strings directly into the **Price Agreements** plugin interface.
* **&#57;03000 Block:** Injects text strings directly into the **Multimedia** plugin interface.
* **&#57;06000 Block onward:** Maps translation strings directly into external **Reportman Report Engine** templates and printed document layout layouts.

## &#49;000000 Block: Welcome Screen, Credits \& Country Mapping

This block maps the initial application splash environment and global country properties. To localize your software environment correctly, adapt the following mapped records:

* **Label ID 1000000 (File Header \& Identification)**: **Source Text**: "English Master File for English South Africa (en-ZA) - Do Not Translate. To customise this language file, create a copy." **Action:** Change this text to describe your localized target country, target industry variant, or to display your translation team's development credits.
* **Label ID 1000001 (Title Bar)**: **Source Text**: "Welcome to \<@APPNAME@\>" Action: Localize this string to control what text populates the application's operating system title bar when the Welcome screen environment launches.
* **Label ID 1000004 (Main Branding Tag)**: **Source Text**: "\<@APPNAME@\>" **Action:** Ensure the core dynamic placeholder tag remains fully operational alongside your localized greeting text edits.

# Global Currency Configuration Mapping

To successfully map your regional currency across the interface and financial print engines, you must coordinate your edits across these specific ID targets:

* **Label ID 1000003 (Physical Currency Symbol)**: **Source Text**: "R" **Action:** Swap this placeholder letter with your region's actual currency symbol (e.g., replacing the South African Rand **R** with **$**, **€**, **£**, **¥**, etc.).
* **Label ID 157 (Primary Currency Word Naming)**: **Location**: Mapped in the main interface text section. **Action:** Input your country's main currency denomination text in words (e.g., Rand, Dollars, Euros).

🖨️ **Print Engine Behavior:** This text is dynamically called when printing documents like **Receipts** and **Payments** where the system must spell out the legal tender value in full written words (e.g., converting a numeric total into "One Hundred and Fifty **Dollars**").

* **Label ID 158 (Fractional Sub-Denomination Word Naming)**: **Location:** Mapped in the main interface text section. **Action:** Input your country's fractional or sub-unit currency name in words (e.g., Cents, Pence).

🖨️ **Print Engine Behavior:** Used by the print engine to finish spelling out the fractional cents or decimal remainder of a text-converted payment total (e.g., completing the written line as "...and Twenty-Five **Cents**").

# Instructions for Translators and Localization Teams

When utilizing English-Journal.dfm or English-Journal-IFRS.dfm as your master translation source text, adhere strictly to the following linguistic rules within the **4,072 string matrix**:

* **Financial \& Bookkeeping Contexts:** All references to financial processing have been explicitly migrated from "Batch" terms (e.g., *Batch Type, Batch Posting*) to "Journal" equivalents (*Journal Type, Journal Posting*). You must translate the word **"Journal"** into your target language.

  * ***Example**:* In German localizations, the term **"Journaltyp"** is now preferred over the legacy, ambiguous **"Stapelart"**.

* **Operational Isolation:** Avoid any translation of financial terms that implies inventory, warehousing, manufacturing, or logistics.
* **The IFRS Matrix Target:** If you are localizing for modern academic environments or international corporate accounting, use English-Journal-IFRS.dfm. This aligns your translation column to standardized international terminology from day one:

| **Structural Area** | **Legacy English Baseline (English.dfm)** | **Journal Baseline (English-Journal.dfm)** | **Modern IFRS Baseline (English-Journal-IFRS.dfm)** |
| --- | --- | --- | --- |
| **Transaction Processing** | Batch | **Journal** | **Journal** |
| **Receivables** | Debtors | Debtors | **Customers** |
| **Payables** | Creditors | Creditors | **Suppliers** |
| **Logistics / Warehouse** | Stock | Stock | **Inventory** |

## Source Preparation (The Baseline Swap)

Before translating, you must explicitly choose your target linguistic alignment and rename the master reference file in your environment directory so the software reads the correct English definitions as the source template:

* **The Legacy Option:** Rename English.dfm to English-batch.dfm. (Batch, Debtors, Creditors, Stock)
* **The Modern Standard Option:** Rename English.dfm to English-jounal.dfm. (Journal, Debtors, Creditors, Stock)
* **The Academic / IFRS Option:** Rename English.dfm to English-journal-IFRS.dfm.&nbsp; (Journal, Customers, Suppliers, Inventory)

**Why this is mandatory:** Swapping these files before beginning localization forces the system to inject clean, pre-isolated source strings (such as "Journal" instead of "Batch") directly into the core **ENGTEXT** column on the **Customise Languages** screen.

This column provides the absolute baseline data stream for your entire localization project. By securing a clear, context-accurate English foundation in the spreadsheet matrix from the very start, you prevent human translators, automated translation engines, and AI language models from misinterpreting financial processing runs as warehouse inventory lots—ensuring clean, professional results with zero manual backtracking.
